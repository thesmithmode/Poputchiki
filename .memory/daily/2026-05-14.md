### Session (14:45)

**Context:** Диагностика и фикс production auth failure в Poputchiki (Telegram Mini App)

**Key Exchanges:**
- Деплоили диагностический билд — `telegramAuth()` стал возвращать точный текст ошибки вместо generic
- Скриншот показал: `API 401: auth failed (initData: 580ch)` — initData не пустой, значит HMAC прошёл
- "auth failed" идёт из catch блока `sql.begin`, не из HMAC verifier — значит транзакция в БД упала

**Decisions Made:**
- Антон дал разрешение на deploy без подтверждения пока идёт отладка: "ДА, не спрашивай каждый раз пока мы только отлаживаем" — squash dev→main теперь без запроса до окончания дебаг-сессии

**Lessons Learned:**
- `INSERT INTO nonces ... ON CONFLICT DO NOTHING RETURNING hash` требует SELECT privilege — `poputchiki_app` имел только INSERT → `permission denied` → catch → "auth failed". Решение: убрать `RETURNING`, проверять через `result.count === 0`
- Диагностика auth: если initData не пустой и статус 401 — смотреть DB транзакцию, а не HMAC
- API Docker logs могут быть 0 байт (pino stdout не захватывается в Docker log-файл) — диагностировать через временный UI-exposed error string

**Action Items:**
- После успешного деплоя фикса: откатить `telegramAuth()` к чистой error handling (убрать diagnostic error format)
- Проверить все остальные таблицы где `poputchiki_app` использует `RETURNING` — возможно аналогичная проблема
