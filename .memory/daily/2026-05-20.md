## Сектор DB — результаты ревью

### Критические баги
1. **Race condition: joinTrip** — `trips` таблица: проверка `passengers_count < max_passengers` и `UPDATE trips SET passengers_count = passengers_count + 1` — два отдельных SQL без `SELECT ... FOR UPDATE`. При 50k concurrent → переполнение.
   **Fix:** Атомарный UPDATE с WHERE-guard:
   ```sql
   UPDATE trips SET passengers_count = passengers_count + 1
   WHERE id = $1 AND passengers_count < max_passengers
   RETURNING id;
   -- если 0 rows → reject
   ```

2. **Дублирующие поездки** — нет UNIQUE constraint на `(driver_id) WHERE status IN ('active','started')`. Два одновременных INSERT могут создать две активные поездки.
   **Fix:**
   ```sql
   CREATE UNIQUE INDEX trips_driver_active_uniq
   ON trips(driver_id)
   WHERE status IN ('active','started');
   ```

3. **location_history bloat** — `location_history` пишется каждые 5с без partition/TTL. За сутки 50k users × 17280 строк = 864M строк.
   **Fix:** Range partitioning по `created_at` (weekly) + `pg_partman` для auto-drop старых партиций.

   **Fix:** `set_config('app.current_user_id', uid, true)` → local = true, сбрасывается при END/ROLLBACK. Или явный `RESET app.current_user_id` в error handler.
