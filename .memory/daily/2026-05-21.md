**User:** <task-notification>**Context:** Squash merge dev→main, prod deploy pipeline — deploy упал дважды, оба раза чинили и перезапускали; финальный deploy 26204568834 в очереди.

**Key Exchanges:**
- Первый deploy упал: `verify-ci` проверял parent SHA = previous main tip (PR squash без CI), а не dev tip. Исправлено в dev, squash повторён.
- Второй deploy упал: `pg_dump` из хоста не видит hostname `postgres` — контейнерная сеть недоступна с хоста. Решение: detection-based `backup-db.sh` — внутри контейнера cron `pg_dump` работает напрямую, с хоста — через `docker exec`.
- Убрана установка `pg_dump` на хост в `deploy.sh` — больше не нужна.
- Кратко обсудили нагрузочную ёмкость: 7GB/8vCPU VPS реалистично 10-20к одновременных активных при PgBouncer; до 50к нужен горизонтальный scaling или апгрейд железа. Нагрузочных тестов (k6/artillery) не делали — белая зона.

**Decisions Made:**
- `backup-db.sh` — detection-based: если внутри контейнера (`POSTGRES_HOST` или env cron), используем прямой `pg_dump`; если с хоста — `docker exec postgres pg_dump`. Убирает зависимость от `pg_dump` на хосте.
- Коммиты только в dev, прямые коммиты в main запрещены — нарушение было поймано и откачено.

**Lessons Learned:**
- `verify-ci` должен проверять SHA dev-ветки, не родительский SHA squash-коммита в main — иначе находит коммит без CI и блокирует deploy.
- `pg_dump` внутри `backup-db.sh` вызывается из cron-контейнера — хост не нужен, переусложнение создало проблему.
- Docker Compose hostname `postgres` доступен только внутри сети контейнеров; с хоста нужен `docker exec` или `localhost:5432` (если порт проброшен).
