**Context:** Оптимизация производительности TMA — bundle size, кэширование, UX загрузки

**Key Exchanges:**
- Пользователь остановил autoresearch после серии мелких экспериментов (<1%) и потребовал оценивать потенциал ДО эксперимента, браться только за >2%
- Пользователь уточнил: реальный эффект vs метрика "2% bundle" — оказалось сделано значительно больше
- Поднят вопрос UX-обмана: кэшированная лента показывает устаревшие поездки, пользователь видит места — на деле заняты. Признано багом безопасности данных
- Пользователь предложил конкретный дизайн индикатора: мини-плашка сверху, значок обновления кликабельный, время актуальности ленты
- Открытый вопрос: автообновление ленты при открытом экране — частота, нагрузка на БД при 2000 одновременных юзерах, есть ли кэш запросов к БД

**Decisions Made:**
- `staleTime` rides: 60s → 20s (кэш устаревает быстрее, снижает риск устаревших данных)
- Экран детали поездки (`useRide`) — всегда свежий запрос, кнопка записи безопасна
- Индикатор обновления ленты — плашка сверху с кликабельным значком и временем актуальности

**Lessons Learned:**
- `stale-while-revalidate` в ленте поездок создаёт UX-ловушку: пользователь видит "доступные" места которых нет — особенно опасно для кнопки "Подать заявку"
