## Key Points

- Stale-while-revalidate кэш ленты поездок показывает свободные места, которые уже заняты
- Пользователь нажимает «Записаться» → ошибка или превышение мест в поездке
- Признано багом безопасности данных, а не UX-багом
- Исправление: `staleTime` ленты `60_000` → `20_000` мс, `useRide` детали → `staleTime: 0`

## Details

TanStack Query по умолчанию использует stale-while-revalidate: отдаёт кэшированные данные мгновенно, а фоновый рефетч обновляет их. При `staleTime: 60_000` окно расхождения составляет до минуты. За это время другой пользователь может занять последнее место в поездке — но лента всё ещё покажет место свободным. Пользователь, видя «1 место», нажимает «Записаться» и получает либо серверную ошибку (409 Conflict), либо попадает в поездку с превышением capacity.

Принципиальное разграничение: лента поездок — информационный контекст (допускает небольшое устаревание), кнопка «Записаться/Отменить» — транзакционное действие (должна отражать актуальное состояние). Поэтому решение асимметрично: снизить `staleTime` ленты до `20_000` мс (снижает окно расхождения, но не убирает нагрузку), а хук `useRide` на экране деталей установить в `staleTime: 0` (всегда свежий запрос перед действием). Агрессивный `staleTime: 0` на всю ленту при 50k DAU создал бы избыточную нагрузку на БД.
