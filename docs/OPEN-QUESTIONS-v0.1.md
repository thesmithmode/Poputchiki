# Открытые вопросы и слепые зоны — Poputchiki v0.1

**Дата:** 2026-05-01
**Контекст:** черновик PRD/SPEC v0.1 опирается на ряд предположений. Этот документ — список того, что нужно подтвердить перед заморозкой v1.0 и стартом Ralph-цикла.

> Формат: `[категория] Вопрос → текущее предположение → почему критично → что решает.`

## A. Продукт и сообщество

### A.1 [онбординг] Кто допускается в приложение?
**Предположение v0.1:** любой Telegram-пользователь с валидным `initData` создаёт аккаунт и получает доступ к ленте.
**Альтернативы:**
- Только участники конкретного TG-чата ЖК Царёво (проверяем через bot, который состоит в чате и читает member list).
- Инвайт-код от уже верифицированного жителя.
- Открытая регистрация + ручное подтверждение админом по номеру квартиры.

**Почему критично:** определяет модель доверия. Открытая регистрация → выше риск мошенников; закрытая → ниже органический рост.

**Нужно решение:** какая степень закрытости приемлема в MVP.

---

### A.2 [роли] Как защититься от фейк-водителей?
**Предположение v0.1:**
- Любой может опубликовать поездку, но не-верифицированный → 1 активная поездка max + жёлтая метка + ниже в выдаче.
- Верификация = заявка с фото пропуска / квартирой / приглашение от верифицированного жителя.

**Альтернативы:**
- Без верификации публикация запрещена (только поиск).
- Двухсторонняя верификация (пассажир тоже верифицирует).
- Bond/депозит — водитель блокирует N₽ как гарантию (отдельная сущность кошелька).

**Почему критично:** пользователь явно назвал «не наебать» как top-priority боль.

**Нужно решение:** какой механизм верификации MVP.

---

### A.3 [ЖК] Один ЖК или мульти-tenant?
**Предположение v0.1:** один ЖК (Царёво), мульти-tenant архитектура заложена в схему (`tenant_id` колонка добавится в фазе 2), но UI один.

**Альтернатива:** изначально мульти-tenant с переключателем сообществ.

**Почему критично:** влияет на схему БД (ranged FK, RLS-политики).

**Нужно решение:** оставляем заглушку или сразу мульти-tenant?

## B. Auth и SSL

### B.1 [SSL] Self-hosted сервер — как раздаём TLS?
**Предположение v0.1:** деплой на Railway, который даёт SSL автоматически.
**Контекст:** пользователь упомянул возможность хостинга на личном сервере; «у меня там и так захочется куча всего, и он довольно медленный».

**Если self-hosted:**
- Caddy с автоматическим Let's Encrypt по DNS-01 challenge (нужен домен).
- Traefik + Let's Encrypt HTTP-01 (нужны открытые 80/443).
- Cloudflare Tunnel — бесплатно, без проброса портов, TLS на стороне CF.
- Tailscale Funnel — для приватного доступа.

**Почему критично:** Telegram WebApp работает ТОЛЬКО по HTTPS с валидным сертификатом, без исключений.

**Нужно решение:** Railway или self-hosted? Если self-hosted — есть ли домен, какой proxy?

---

### B.2 [auth] Supabase Auth via Telegram — кастомный JWT или magic link?
**Предположение v0.1:** наш backend верифицирует HMAC, выдаёт Supabase-совместимый JWT, подписанный JWT secret Supabase. Фронт использует этот JWT для RLS-чтений.

**Альтернатива:** Supabase Edge Function `sign-in-with-telegram` — переносит логику ближе к БД, но сложнее тестировать локально.

**Почему критично:** определяет точку доверия. Кастомный JWT — наш HMAC-код становится TCB; Edge Function — хуже наблюдаемость.

**Нужно решение:** где живёт логика верификации.

---

### B.3 [identity-guard] Cookie или localStorage?
**Предположение v0.1:** cookie `tg_uid` HttpOnly=false по эталону Telegram-export-clean.

**Альтернатива:** localStorage + JS check (в Telegram WebApp куки и так часто ломаются между сессиями).

**Почему критично:** Telegram WebApp на iOS/Android по-разному обрабатывает куки в WebView; на iOS бывали проблемы с SameSite.

**Нужно решение:** проверить экспериментально на iOS/Android, может локалстораджа достаточно.

## C. Данные и приватность

### C.1 [ПД] Что считаем персональными данными и шифруем?
**Предположение v0.1:** телефон и номер квартиры → pgcrypto при хранении.
**Не зашифровано:** имя, TG username, аватар.

**Вопрос:** ФЗ-152 в РФ — нужен ли уведомительный учёт оператора ПД для приватного MVP?

**Нужно решение:** консультация юриста или явное «MVP приватный, без официального оформления, на свой риск».

---

### C.2 [бэкапы] Куда бэкапим?
**Предположение v0.1:** S3-совместимое хранилище (Backblaze / Yandex Object Storage).

**Альтернативы:** свой Hetzner Storage Box, Google Cloud Storage, локальный сервер заказчика.

**Почему критично:** заказчик подчеркнул бэкапы и сжатие. Нужен реальный bucket с реквизитами.

**Нужно решение:** провайдер хранилища + бюджет.

---

### C.3 [ретеншн] Сколько хранить отменённые поездки и истёкшие отзывы?
**Предположение v0.1:** активные поездки — навсегда, отменённые — 90 дней, отзывы — навсегда.

**Альтернатива:** soft delete с TTL для всего, кроме отзывов.

**Почему критично:** влияет на размер БД и восприятие приватности.

**Нужно решение:** политика ретеншна — какой default.

---

### C.4 [GDPR-like] Право на удаление аккаунта?
**Предположение v0.1:** есть, удаление каскадное; но отзывы анонимизируются (`passenger_id = NULL`), не удаляются — иначе ломается история водителя.

**Нужно решение:** подтвердить или предложить альтернативу.

## D. Архитектура и инфраструктура

### D.1 [Redis или нет?] Где живут nonce и rate-limit?
**Предположение v0.1:** в Supabase (отдельная таблица `nonces` + индекс по `expires_at`, периодическая очистка). Rate-limit — Hono in-memory + sticky session на Railway.

**Альтернативы:**
- Upstash Redis (managed) — для надёжного TTL и атомарных setIfAbsent.
- Свой Redis в Railway addon.

**Почему критично:** in-memory rate-limit не работает при горизонтальном масштабировании; Supabase для nonce — лишний round-trip.

**Нужно решение:** ставить Redis сразу или жить на Supabase до боли.

---

### D.2 [realtime] Supabase Realtime достаточен?
**Предположение v0.1:** да, channel `rides:active` — публикуем INSERT/UPDATE/DELETE rides.

**Риски:** Supabase Realtime платный по сообщениям; при 1000+ пользователей и активной ленте — счёт может пойти в десятки $.

**Альтернатива:** SSE через Hono / WebSocket свой.

**Нужно решение:** оценить trafic projection и принять.

---

### D.3 [TG bot для notify] — отдельный bot или MiniApp Bot?
**Предположение v0.1:** один и тот же bot, через который запускается MiniApp, шлёт push-сообщения через `sendMessage` API.

**Риск:** если bot заблокирован пользователем — пуши не доходят, нужно отслеживать `bot_blocked`-событие.

**Нужно решение:** подтвердить + добавить обработку в audit-log.

---

### D.4 [рейтинг — формула] Простой avg или взвешенный?
**Предположение v0.1:** простое среднее по всем reviews.

**Альтернатива:** Bayesian average, чтобы 1 отзыв 5* у новичка не давал ему top-rank.

**Почему критично:** простой avg → манипуляции (друг написал 5*).

**Нужно решение:** какую формулу включаем в MVP.

## E. Автономный цикл разработки

### E.1 [ralph.sh] Адаптируем существующий или переписываем?
**Предположение v0.1:** существующий `saas/saas-project-ralph/ralph.sh` адаптируем для TS-проекта (заменим `uv run pytest` → `bun test`, `uv run ruff` → `bun lint`), внедрим минимум из `Доработать_скрипт.md`:
- timeout per iteration (20 мин);
- логи итераций в `logs/iteration-N.txt`;
- backup commit перед каждой итерацией;
- pre-flight check (зависимости, env, чистое дерево);
- coverage gate в test step.

**Альтернатива:** написать с нуля под Bun runner с UI-прогрессом.

**Нужно решение:** объём работ по скрипту в фазе bootstrap.

---

### E.2 [task сегментация] Какова целевая длина одной задачи в `tasks.json`?
**Предположение v0.1:** 30 минут агентского времени = одна задача. Сложные фичи разбиваются на цепочку зависимых задач.

**Нужно решение:** подтверждено пользователем в шаблоне `prompt_от_PRD_до_tasks.md`.

---

### E.3 [тесты] Что блокирует «зелёный pipeline»?
**Предположение v0.1:** lint + typecheck + unit (≥90%) + integration + E2E + coverage-gate. Mutation тесты — еженедельно в nightly, не блок.

**Альтернатива:** mutation в каждой PR — слишком медленно.

**Нужно решение:** подтвердить или ослабить.

---

### E.4 [secrets для Claude в Ralph] Где живут API keys для CI?
**Предположение v0.1:** Claude вызывается локально на машине заказчика (Anton), keys в env. CI на GitHub Actions использует только service tokens (Supabase, Railway).

**Альтернатива:** деплой ralph.sh в облако (LangServe / agent-sandbox) — пока избыточно для одного проекта.

**Нужно решение:** подтверждено по умолчанию.
