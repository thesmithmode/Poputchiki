## 1. Обзор продукта

### 1.1 Описание
Telegram MiniApp для жителей ЖК «Царёво», заменяющий ручной поиск попутчиков в Telegram-группе. Пользователи публикуют поездки (как водители) и находят попутчиков (как пассажиры) через структурированный список с фильтрами, рейтингом, отзывами и историей регулярных рейсов.

### 1.2 Проблема
Существующая Telegram-группа «Попутчики Царёво»:
- сообщения теряются в скролле; поиск только по тексту;
- нет рейтингов и отзывов о водителях;
- регулярные рейсы (например, «каждый будний день в 09:00 в центр») не выделены как сущность;
- нет фильтров по времени / направлению / цене / количеству мест;
- нет защиты от мошенников, выдающих себя за водителей.

### 1.3 Цели
- **Скорость поиска**: пассажир за ≤30 секунд находит подходящую поездку.
- **Доверие**: рейтинг + отзывы + история водителя видны до контакта.
- **Регулярность**: повторяющиеся рейсы как первоклассная сущность.
- **Безопасность данных**: ноль утечек ПД, регулярные шифрованные бэкапы, RLS на каждом запросе.
- **Автономная разработка**: продукт строится Claude Code в Ralph-цикле; PRD — единственная точка истины фич.

### 1.4 North Star Metrics
- **WAU / MAU** — активные жители (один ЖК, поэтому не DAU).
- **Match rate** — доля поездок с ≥1 подтверждённым попутчиком.
- **Time to match** — медианное время от публикации поездки до первого контакта пассажира.
- **Driver trust score** — медианный рейтинг активных водителей (>4.5 — здоровое сообщество).
- **Repeat rate** — доля пользователей, использующих приложение ≥1 раз в неделю.

### 1.5 Не-цели MVP (явно вне scope)
- Платежи внутри приложения (СБП / TG Stars / крипта) — пост-MVP.
- Расширение на другие ЖК — пост-MVP (мульти-tenant архитектура заложена, но UI один ЖК).
- Голосовые сообщения / AI-ассистент / Whisper — отложено до подтверждённого use-case (в стеке заложено, но в MVP не используется).
- Оценки пассажиров водителями — фаза 2 (асимметричный рейтинг сначала).
- Карта / геокодирование точек подачи — пост-MVP (MVP оперирует именованными направлениями).

## 2. Целевая аудитория

### 2.1 Сообщество
Жители ЖК «Царёво» (Казань). Закрытый онбординг — только участники Telegram-группы (см. вопрос об инвайт-механике в open-questions).

### 2.2 Persona 1: Водитель «Регулярщик Игорь»
- Каждое утро 08:30 едет в офис в центр.
- Готов взять 1–2 попутчиков за 150–200 руб.
- Хочет один раз настроить расписание, не публиковать заново каждый день.
- Боль: выгорание от ручных постов в группе.

### 2.3 Persona 2: Пассажир «Утренняя Лена»
- Опаздывает, нужна поездка «прямо сейчас» в сторону метро.
- Хочет видеть кто едет в ближайшие 30–60 минут, по какой цене, сколько мест.
- Боль: листает группу и не понимает, актуально ли сообщение часовой давности.

### 2.4 Persona 3: Эпизодик «Вечерний Дима»
- Иногда нужно в аэропорт / на вокзал / по делам поздно вечером.
- Готов и сам подвезти, и поехать с кем-то — зависит от ситуации.
- Боль: не помнит, кто из водителей ездит в аэропорт; нужен фильтр.

## 3. Ролевая модель и сценарии

### 3.1 Модель ролей (решение для v0.1)
**Унифицированный кабинет** — каждый зарегистрированный пользователь может одновременно публиковать поездки (driver-mode) и откликаться на чужие (passenger-mode). Роль определяется не пользователем, а контекстом действия.

**Дополнительный флаг `is_verified_driver`** — отдельный статус, выдаваемый только после прохождения верификации:
- подтверждение что человек реально живёт в ЖК (см. open-questions: фото пропуска / номер квартиры / приглашение от уже верифицированного жителя);
- опционально — подтверждение водительского удостоверения и фото авто.

**Анти-абьюз**:
- Без `is_verified_driver` водитель может публиковать поездки, но они помечены жёлтой меткой «не верифицирован» и стоят ниже в выдаче.
- Лимит на количество активных поездок у не-верифицированного: 1.
- Жалобы от других жителей понижают рейтинг; >N жалоб → авто-блок до ручной проверки.

### 3.2 Ключевые user flows

**Flow A: онбординг**
1. Житель открывает MiniApp по ссылке из Telegram-группы.
2. Telegram передаёт `initData`; backend верифицирует HMAC + `auth_date` + nonce; создаёт/находит юзера в Supabase.
3. Первый вход: короткий онбординг — имя (предзаполнено из TG), номер квартиры, телефон (опционально, для контакта вне TG), согласие на ПД.
4. Главный экран — лента активных поездок.

**Flow B: водитель публикует разовую поездку**
1. Водитель → «Создать поездку» → форма:
   - дата + время (по умолчанию ближайшие 30 минут);
   - направление (выпадающий список + свободный ввод; нормализация на бэке);
   - цена (число или «договорная»);
   - количество мест (1–4);
   - комментарий (текст ≤ 200 символов).
2. Submit → запись в БД → лента обновлена через realtime (Supabase Realtime).

**Flow C: водитель публикует регулярный рейс**
1. Та же форма + чекбокс «Регулярный».
2. Открываются поля: дни недели (Пн-Вс мультиселект), исключения (даты), окно действия (с / по).
3. Backend разворачивает шаблон в materialized view ближайших 14 дней (cron каждую ночь).

**Flow D: пассажир ищет поездку**
1. Лента активных поездок — отсортирована по времени отправления.
2. Фильтры: направление, диапазон времени, цена ≤ X, мест ≥ N, только верифицированные водители, только избранные водители.
3. Тап на поездку → детальная карточка → профиль водителя (рейтинг, отзывы, история).
4. Кнопка «Написать в Telegram» → `tg://user?id={driver_tg_id}` → выходит в чат с водителем.
5. Кнопка «Откликнуться» → запись `ride_request` (для метрик match rate, водитель видит уведомление в MiniApp).

**Flow E: завершение поездки + отзыв**
1. Через N часов после `departure_at` → водитель получает push (через Telegram bot API): «Поездка состоялась? Кого подвозил?».
2. Водитель отмечает фактических пассажиров из списка откликнувшихся.
3. Пассажирам уходит push с просьбой оставить отзыв (1–5 звёзд + текст ≤ 300 символов).
4. Отзыв публичный, привязан к driver_id; private-комментарий пассажира к водителю — отдельная таблица, видна только автору.

**Flow F: избранное и приватные комментарии**
- Любой пользователь может добавить водителя в избранное (свой список).
- Личный комментарий к карточке водителя («Возит спокойно, не курит» — для себя) — отдельное поле, не публикуется.

## 4. Функциональные требования (MVP scope)

### 4.1 F-AUTH: Аутентификация через Telegram initData
- Только через Telegram MiniApp. Web-форма входа отсутствует.
- HMAC-SHA256 верификация `initData` (по протоколу Telegram WebApp).
- Окно `auth_date`: ±5 минут.
- Replay-защита: nonce по `hash` в Redis/Supabase KV, TTL = окну.
- Identity-guard cookie + клиентская проверка против `Telegram.WebApp.initDataUnsafe.user.id`.
- Fail-closed: при недоступности nonce-стора → отказ в логине.
- Эталон: `C:\Soft\Projects\Telegram-export-clean\src\main\java\com\tcleaner\dashboard\auth\telegram\` (портировать на Hono middleware).

### 4.2 F-RIDE: Поездки
- Модели: `ride`, `ride_template` (для регулярных), `ride_request` (отклики).
- CRUD: создание, редактирование (до `departure_at`), отмена (с уведомлением откликнувшимся).
- Список: сортировка по времени, фильтры (направление / время / цена / места / верификация / избранное).
- Realtime обновление списка (Supabase Realtime channel).

### 4.3 F-PROFILE: Профиль и рейтинг
- Карточка водителя: имя, аватар (TG photo url), статус верификации, рейтинг (avg + count), история (всего поездок, доля состоявшихся), отзывы (paginated).
- Личный кабинет: список своих поездок (активные / архив), черновики шаблонов, избранное.

### 4.4 F-REVIEW: Отзывы
- 1–5 звёзд + текст ≤ 300 символов.
- 1 отзыв на (passenger_id, ride_id).
- Запрет редактирования через 24 часа после публикации.
- Жалоба на отзыв → модерация (см. open-questions: автомод/ручной).

### 4.5 F-FAVORITE: Избранное
- Подписка пользователя на водителя.
- Уведомление при публикации новой поездки избранного (опционально, чекбокс в настройках).
- Приватный комментарий к водителю — текстовое поле, видно только автору.

### 4.6 F-NOTIFY: Уведомления
- Push через Telegram Bot API (отдельный bot аккаунт, не сам MiniApp).
- Триггеры: новый отклик на твою поездку, новое сообщение от пассажира, отмена твоей поездки (если ты отклик), запрос на отзыв через N часов.
- Настройки: глобальный mute, выбор категорий.

### 4.7 F-ADMIN: Администрирование
- Один владелец-админ (Anton) — захардкожен по `tg_id` через env.
- Дашборд: список пользователей, поездки в реалтайме, отзывы на модерации, кнопки `ban/unban/verify`.
- Аудит-лог админ-действий.

## 5. Нефункциональные требования

### 5.1 Безопасность (top priority)
- **RLS на ВСЁМ**: ни одна таблица Supabase не доступна без RLS-политики.
- **JWT**: backend выдаёт Supabase-совместимый JWT с custom claim `tg_id` после Telegram-верификации.
- **Шифрование персональных данных**: телефон / номер квартиры — pgcrypto или application-level AES (см. open-questions).
- **Бэкапы**: ежедневный snapshot Supabase + еженедельный full dump в внешнее хранилище (S3-совместимое), сжатие zstd, шифрование at-rest. Тест восстановления раз в месяц (cron job).
- **Аудит-лог**: каждое state-changing API-действие → таблица `audit_log` (actor_id, action, target_id, payload_hash, ip, ts).
- **Rate limiting**: 100 req/min на user-id, 1000 req/min на ip; 429 при превышении.
- **Валидация на входе**: zod-схемы на КАЖДОМ Hono endpoint; ошибки → 422 + structured payload.
- **Secrets**: только в env Railway / Supabase Vault. Никогда в репо. Pre-commit hook сканирует на утечки (gitleaks).
- **HTTPS-only**: Railway даёт SSL из коробки; для self-hosted — отдельный план в open-questions.
- **CSP + HSTS + X-Frame-Options**: фронт за Hono + CDN с правильными заголовками.

### 5.2 Производительность
- p95 API < 200ms на чтение, < 500ms на запись.
- Лента поездок: индекс на `(departure_at, status)`; LIMIT 50 + cursor pagination.
- N+1 запрещён: review-агрегаты — materialized view `driver_stats`, refresh каждые 5 минут.
- Realtime: WebSocket через Supabase, fallback на polling раз в 30 сек.

### 5.3 Надёжность
- Health-check endpoints `/health` и `/readiness`.
- Graceful shutdown Hono + Bun (drain in-flight requests).
- Idempotency keys на write-endpoints (header `Idempotency-Key`).
- Circuit breaker для вызовов в Telegram Bot API (при 5xx — пауза 30 сек).
- Retry с exponential backoff на DB connection errors.

### 5.4 Совместимость
- Telegram WebApp API ≥ 6.0 (актуальная версия на 2026-05).
- Браузеры: WebView Telegram (Android/iOS/Desktop). Standalone web — не цель.
- Темизация: подхват `Telegram.WebApp.colorScheme` (light/dark).

### 5.5 Доступность
- WCAG 2.1 AA на критичных сценариях (создание поездки, поиск, отзыв).
- Контраст / aria-labels / клавиатурная навигация в desktop-WebApp.

### 5.6 Локализация
- v1.0 — только русский. i18n-ready через `i18next` (заложить структуру).

## 6. Технический стек (утверждённый)

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| Backend | TypeScript + Hono + Bun | Лёгкий, быстрый старт; нативная поддержка Web Standards (Request/Response); Bun даёт sub-100ms cold start |
| Frontend | TypeScript + Vite + React (SPA) | Утверждено заказчиком; Vite быстрый dev cycle; React — экосистема Telegram WebApp libs (`@telegram-apps/sdk-react`) |
| БД | Supabase (PostgreSQL) | Утверждено; даёт Auth, RLS, Realtime, Storage, Edge Functions из коробки |
| Деплой | Railway (один контейнер) | Утверждено; SSL/CD из коробки; альтернатива самохостинга — open-question |
| AI чат | Anthropic Claude Sonnet | Заложено на пост-MVP (например, ассистент-помощник пассажира) |
| Транскрипция | OpenAI Whisper | Заложено на пост-MVP (голосовое описание поездки) |
| Auth | Supabase Auth + Telegram initData | Custom JWT после HMAC-проверки |
| Аналитика | PostHog (опционально) | Self-hosted или cloud — решить позже |

См. `docs/SPEC-Architecture-v0.1.md` для деталей.

## 7. Метрики, мониторинг, логирование

- **Метрики**: PostHog product events (`ride_created`, `ride_match`, `review_left`, `verification_passed`).
- **Логи**: structured JSON через `pino`, агрегация в Railway logs / Loki (open-question).
- **Tracing**: OpenTelemetry → Tempo / Honeycomb (опционально, фаза 2).
- **Алерты**: error rate > 1%, p95 latency > 1s, Supabase quota > 80% — Telegram-бот алертов админу.
