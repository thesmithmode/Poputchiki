## 1. Обзор продукта

### 1.1 Описание
Telegram MiniApp для жителей ЖК «Царёво», заменяющий ручной поиск попутчиков в Telegram-группе. Пользователи публикуют поездки и находят попутчиков через структурированный список с фильтрами, рейтингом, отзывами, лайками и историей регулярных рейсов. Карта показывает точки и маршруты.

**Юридическая позиция продукта**: площадка для попутчиков (по модели BlaBlaCar), не агрегатор такси. Сервис не заключает договор перевозки и не является оператором; только обмен предложениями и контактами между жителями.

### 1.2 Проблема
Существующая Telegram-группа «Попутчики Царёво»:
- сообщения теряются в скролле; поиск только по тексту;
- нет рейтингов и отзывов о водителях;
- регулярные рейсы (например, «каждый будний день в 09:00 в центр») не выделены как сущность;
- нет фильтров по времени / направлению / цене / количеству мест;
- нет защиты от мошенников и ботов.

### 1.3 Цели
- **Скорость поиска**: пассажир за ≤30 секунд находит подходящую поездку.
- **Доверие**: рейтинг, отзывы, лайки, история водителя видны до контакта.
- **Регулярность**: повторяющиеся рейсы как первоклассная сущность.
- **Безопасность данных**: ноль утечек ПД, ежедневные локальные бэкапы с тестом восстановления, RLS на каждом запросе, deny-by-default везде.
- **Threat model**: каждый пользователь — потенциальный атакующий.
- **Бюджет $0**: MVP полностью на бесплатных тирах. Платные сервисы — только после введения подписки.
- **Автономная разработка**: продукт строится Claude Code в Ralph-цикле; PRD — единственная точка истины фич.

### 1.4 North Star Metrics
- **WAU / MAU** — активные жители (один ЖК, поэтому не DAU).
- **Match rate** — доля поездок с ≥1 подтверждённым попутчиком.
- **Time to match** — медианное время от публикации поездки до первого контакта пассажира.
- **Trust score** — медианное число лайков активного пользователя (>3 — здоровое сообщество).
- **Repeat rate** — доля пользователей, использующих приложение ≥1 раз в неделю.

### 1.5 Не-цели MVP (явно вне scope)
- **Платежи внутри приложения** (СБП / TG Stars / крипта) — пост-MVP; на текущей фазе деньги вообще не идут через сервис, водитель и пассажир расплачиваются вне приложения.
- **Расширение на другие ЖК** — пост-MVP (мульти-tenant архитектура заложена в схему, но UI один ЖК).
- **Голосовые сообщения / AI-ассистент / Whisper** — отложено до подтверждённого use-case (в стеке заложено, но в MVP не используется).
- **Двусторонний рейтинг** — в SCHEMA закладывается с самого начала, в UI MVP — обе стороны видят и ставят, но детальная страница рейтинга пассажира — фаза 2.
- **Верификация водителя** — MVP не требует верификации; любой авторизованный пользователь может публиковать поездки. Анти-абьюз — через фильтры по возрасту аккаунта и количеству полученных лайков.
- **Ручная модерация отзывов** — фаза 1.5; в MVP жалоба только сохраняется, но не блокирует.

## 2. Целевая аудитория

### 2.1 Сообщество
Жители ЖК «Царёво» (Казань). Регистрация **открытая** — любой пользователь Telegram через `initData`. Закрытость заменяется фильтрами доверия (см. п. 4.7).

### 2.2 Persona 1: Водитель «Регулярщик Игорь»
- Каждое утро 08:30 едет в офис в центр.
- Готов взять 1–2 попутчиков за 150–200 руб.
- Хочет один раз настроить расписание, не публиковать заново каждый день.
- Боль: выгорание от ручных постов в группе.

### 2.3 Persona 2: Пассажир «Утренняя Лена»
- Опаздывает, нужна поездка «прямо сейчас» в сторону метро.
- Хочет видеть кто едет в ближайшие 30–60 минут, по какой цене, сколько мест, на карте.
- Боль: листает группу и не понимает, актуально ли сообщение часовой давности; не понимает кто из водителей надёжный.

### 2.4 Persona 3: Эпизодик «Вечерний Дима»
- Иногда нужно в аэропорт / на вокзал / по делам поздно вечером.
- Готов и сам подвезти, и поехать с кем-то — зависит от ситуации.
- Боль: не помнит, кто из водителей ездит в аэропорт; нужен фильтр и история.

## 3. Ролевая модель и сценарии

### 3.1 Унифицированный кабинет
Каждый зарегистрированный пользователь может одновременно:
- публиковать поездки (driver-mode действие);
- откликаться на чужие (passenger-mode действие);
- ставить лайки и оставлять отзывы.

Роль определяется не пользователем, а контекстом действия. Никаких отдельных «driver-only» / «passenger-only» аккаунтов.

### 3.2 Анти-абьюз через фильтры доверия (без обязательной верификации)
**В MVP отказ от обязательной верификации** — порог входа должен быть низким. Защита от ботов и мошенников строится на:

1. **Фильтры в выдаче ленты** (включает пользователь, не админ):
   - «только аккаунты ≥7 дней назад» (по умолчанию выключен);
   - «только с ≥3 лайками от других участников»;
   - «только избранные»;
   - «только верифицированные» (когда верификация появится — пост-MVP).
2. **Лимиты по дефолту** (без флагов от пользователя):
   - аккаунт младше 24 часов: максимум 1 активная поездка одновременно;
   - аккаунт без единого лайка от других: максимум 3 поездки в сутки;
   - аккаунт младше 7 дней с 0 лайков: жёлтый бейдж «новый участник» рядом с именем.
3. **Жалобы → авто-блок**:
   - 5+ жалоб от ≥5 разных пользователей в течение 7 дней → авто-block до ручной проверки.
   - Жалоба от одного и того же пользователя на одного и того же — лимит 1 в неделю (антиспам жалоб).
4. **Repeat-аккаунты**: tg_id уникален; нельзя «обновиться» через новый аккаунт MiniApp без нового tg-аккаунта в Telegram (а это требует нового телефона).

### 3.3 Симметричные лайки (двусторонний trust)
- **Водитель → пассажир**: после поездки водитель ставит лайк пассажиру (или не ставит).
- **Пассажир → водитель**: после поездки пассажир ставит лайк водителю.
- Один лайк = одна сторона, одна поездка, один таргет. UNIQUE-ограничение `(subject_id, target_id, ride_id)`.
- Лайки **публичны** в виде агрегата (число лайков рядом с именем), отдельные лайки — нет.
- Накрутка: невозможна без реальной поездки и согласия второй стороны (см. п. 3.4 — поездка должна быть подтверждена обеими).

### 3.4 Подтверждение состоявшейся поездки
- В течение 24 часов после `departure_at` водитель отмечает фактических пассажиров из откликнувшихся.
- Пассажир в течение 24 часов получает push с просьбой подтвердить участие. Без подтверждения — поездка считается «не состоявшейся» для целей лайков и отзывов.
- Лайки и отзывы доступны только если поездка подтверждена обеими сторонами.

### 3.5 Ключевые user flows

**Flow A: онбординг**
1. Житель открывает MiniApp по ссылке из Telegram-группы или директ-линка `t.me/<bot>?startapp`.
2. Telegram передаёт `initData`; backend верифицирует HMAC + `auth_date` + nonce; создаёт/находит юзера в Supabase.
3. Первый вход: короткий онбординг — имя (предзаполнено из TG), номер квартиры (опционально), согласие на ПД.
4. Главный экран — лента активных поездок.

**Flow B: водитель публикует разовую поездку**
1. Водитель → «Создать поездку» → форма:
   - дата + время (по умолчанию ближайшие 30 минут);
   - точка отправления (карта/выпадающий список именованных точек ЖК + произвольная точка);
   - точка назначения (карта + список популярных направлений);
   - цена (число или «договорная»);
   - количество мест (1–4);
   - комментарий (текст ≤ 200 символов).
2. Submit → запись в БД (с `from_lat/from_lng/to_lat/to_lng`) → лента обновлена через realtime (Supabase Realtime).

**Flow C: водитель публикует регулярный рейс**
1. Та же форма + чекбокс «Регулярный».
2. Открываются поля: дни недели (Пн-Вс мультиселект), исключения (даты), окно действия (с / по).
3. Backend разворачивает шаблон в materialized view ближайших 14 дней (cron каждую ночь).

**Flow D: пассажир ищет поездку**
1. Лента активных поездок — отсортирована по времени отправления; альтернативное представление — карта (точки + маршрутные линии).
2. Фильтры: направление (ввод/карта), диапазон времени, цена ≤ X, мест ≥ N, доверие (см. п. 3.2.1), избранное.
3. Тап на поездку → детальная карточка → профиль водителя (рейтинг, лайки, отзывы, история, % состоявшихся).
4. Кнопка «Написать в Telegram» → `tg://user?id={driver_tg_id}` → выходит в чат с водителем.
5. Кнопка «Откликнуться» → запись `ride_request` (для метрик match rate, водитель видит уведомление в MiniApp).

**Flow E: завершение поездки + лайк + отзыв**
1. Через N часов после `departure_at` → водитель получает push: «Поездка состоялась? Кого подвозил?».
2. Водитель отмечает фактических пассажиров. Пассажир получает push с подтверждением и опцией оставить лайк/отзыв.
3. Лайк — 1-клик. Отзыв — опционально, 1–5 звёзд + текст ≤ 300 символов.
4. Отзыв публичный, привязан к driver_id; private-комментарий пассажира к водителю — отдельная таблица, видна только автору.

**Flow F: избранное и приватные комментарии**
- Любой пользователь может добавить водителя/пассажира в избранное.
- Личный комментарий к карточке («Возит спокойно, не курит» — для себя) — приватное поле, не публикуется.

**Flow G: жалоба**
- В карточке любой поездки и любого пользователя — кнопка «Пожаловаться».
- Причины: спам / мошенничество / оскорбление / другое (текст).
- В MVP — только запись в `complaints`; модерация автоматическая по правилам в п. 3.2.3.

## 4. Функциональные требования (MVP scope)

### 4.1 F-AUTH: Аутентификация через Telegram initData
- Только через Telegram MiniApp. Web-форма входа отсутствует.
- HMAC-SHA256 верификация `initData` (по протоколу Telegram WebApp).
- Окно `auth_date`: ±5 минут.
- Replay-защита: nonce по `hash`, TTL = окну.
- Identity-guard: server-set cookie + клиентская проверка против `Telegram.WebApp.initDataUnsafe.user.id` + server-side проверка соответствия cookie ↔ JWT при каждом запросе. Любой mismatch → 401 + force re-auth.
- Fail-closed: при недоступности nonce-стора → отказ в логине.
- Эталон: `C:\Soft\Projects\Telegram-export-clean\src\main\java\com\tcleaner\dashboard\auth\telegram\` (портировать на Hono middleware).

### 4.2 F-RIDE: Поездки
- Модели: `ride`, `ride_template` (для регулярных), `ride_request` (отклики), `ride_participation` (подтверждённые участники).
- CRUD: создание, редактирование (до `departure_at`), отмена (с уведомлением откликнувшимся).
- Список: сортировка по времени, фильтры (направление, время, цена, места, доверие, избранное).
- Realtime обновление списка (Supabase Realtime channel).
- Геополя: `from_lat/lng`, `to_lat/lng`, `from_label`, `to_label`.

### 4.3 F-MAP: Карта
- View переключатель «Список / Карта» в ленте.
- Карта показывает точки активных поездок и линии маршрутов (упрощённо — straight line между точками; реальный routing — пост-MVP).
- Тап на маркер → карточка поездки.
- Стек: **Leaflet + OpenStreetMap** (бесплатно, без API-ключей, без квот).
- В будущем: tile cache на нашем edge / fallback на Yandex Maps SDK (бесплатно до 25k запросов/день) если OSM окажется медленным.

### 4.4 F-PROFILE: Профиль и рейтинг
- Карточка пользователя: имя, аватар (TG photo url), счётчик лайков (полученных от других), бейдж «новый участник» если подходит, рейтинг (avg + count), история (всего поездок, доля состоявшихся), отзывы (paginated).
- Личный кабинет: список своих поездок (активные / архив), черновики шаблонов, избранное (двусторонний список — водители и пассажиры).

### 4.5 F-LIKE: Лайки
- 1 лайк = (`subject_id`, `target_id`, `ride_id`). UNIQUE-индекс.
- Доступен только после подтверждения поездки обеими сторонами.
- Удаление: можно отозвать лайк в течение 24 часов после установки.
- Агрегат `likes_received_count` в `users` обновляется триггером после INSERT/DELETE в `likes`.

### 4.6 F-REVIEW: Отзывы
- 1–5 звёзд + текст ≤ 300 символов.
- 1 отзыв на (`subject_id`, `target_id`, `ride_id`). Симметрично — и водитель пишет о пассажире, и пассажир о водителе.
- Запрет редактирования через 24 часа после публикации.
- Жалоба на отзыв → в `complaints`; в MVP без активной модерации.

### 4.7 F-FAVORITE: Избранное
- Подписка пользователя на любого другого (водителя или пассажира).
- Уведомление при публикации новой поездки избранного водителя (опционально, чекбокс в настройках).
- Приватный комментарий — текстовое поле, видно только автору.

### 4.8 F-NOTIFY: Уведомления
**Две ветки** — для пользователей и для админа.

**Пользовательские уведомления** (push через Telegram Bot API, тот же бот что хостит MiniApp):
- категории (см. таблицу `notification_preferences` в SPEC §4):
  - `ride_request` — кто-то откликнулся на твою поездку;
  - `ride_cancelled` — твоя поездка или поездка где ты пассажир отменена;
  - `confirm_participation` — запрос подтверждения после `departure_at`;
  - `like_received` — получил лайк;
  - `review_received` — получил отзыв;
  - `favorite_new_ride` — избранный водитель опубликовал поездку;
  - `support_reply` — ответ от админа на твоё обращение в поддержку;
  - `system` — системные алерты (мут запрещён, например, бан аккаунта).
- Каждая категория — отдельный чекбокс в настройках (выборочное отключение).
- Глобальный mute — кнопка «Отключить все уведомления кроме системных».
- Обработка `bot_blocked`-события → пометка `notify_disabled=true` в `users`.

**Админ-алерты** (отдельный чат `ADMIN_TG_CHAT_ID` в env):
- Технические: error rate >1%, p95 latency >1s, бэкап failed, restore drill failed, Supabase quota >80%.
- Продуктовые: новое обращение в техподдержку, авто-блок пользователя по жалобам, подозрительная активность (10+ поездок от свежего аккаунта за час).
- Не слать спам: thresholds + cooldown 5 мин на одну категорию.

### 4.9 F-SUPPORT: Обращения в техподдержку
- В UI — пункт меню «Поддержка» → форма с полем text (≤ 2000 символов) → submit.
- Запись в таблицу `support_messages` (status='open').
- Админ получает push в `ADMIN_TG_CHAT_ID` с превью и ссылкой на админ-дашборд.
- Админ из дашборда отвечает текстом → запись `reply_text` + push пользователю в категории `support_reply`.
- Пользователь видит свои обращения в личном кабинете (история + статус).

### 4.10 F-FILTERS: Фильтры доверия
- В UI — раздел «Фильтры → Доверие»:
  - чекбокс «только аккаунты старше N дней» (slider 0–30, default 0);
  - чекбокс «только с ≥M лайками» (slider 0–10, default 0);
  - чекбокс «только избранные».
- Применяется к ленте и карте симметрично.
- Сохраняется в localStorage браузера (чтобы не настраивать каждый раз).

### 4.11 F-COMPLAINT: Жалобы
- Кнопка «Пожаловаться» в карточке поездки и пользователя.
- Запись в `complaints` (reporter_id, target_user_id, target_ride_id?, reason_code, text?).
- Авто-блок при пороге (см. 3.2.3).
- Антиспам жалоб: 1 жалоба от одного пользователя на одного и того же таргета в неделю.

### 4.12 F-ADMIN: Администрирование
- Один владелец-админ (Anton) — захардкожен по `tg_id` через env.
- Дашборд (доступен через MiniApp админу):
  - список пользователей с фильтрами (подозрительные / новые / банн);
  - поездки в реалтайме;
  - жалобы (`complaints`) с кнопками resolve/dismiss;
  - **очередь обращений в техподдержку** (`support_messages` open) с возможностью ответить;
  - мониторинг алертов (последние N админ-уведомлений из своей же ленты, для контекста);
  - кнопки `ban/unban/verify(future)`.
- Аудит-лог админ-действий (`audit_log`).

## 5. Нефункциональные требования

### 5.1 Безопасность (top priority — threat model «every user is hostile»)
- **Deny-by-default**: каждая таблица — RLS с явным `USING/WITH CHECK`; новый эндпоинт без явной auth/authz-проверки → fail в CI (lint-rule).
- **RLS на ВСЁМ**: ни одна таблица Supabase не доступна без RLS-политики. Тест `deny_by_default_test` гоняется в CI и проверяет, что без auth токена все select/insert/update возвращают 0 rows.
- **JWT**: backend выдаёт Supabase-совместимый JWT с custom claim `tg_id` после Telegram-верификации.
- **Шифрование персональных данных**: телефон / номер квартиры — pgcrypto `pgp_sym_encrypt` (ключ в env); никогда не в plaintext колонке.
- **Бэкапы**: ежедневный snapshot БД через `pg_dump` локально в папку `./backups/` внутри проекта, сжатие zstd, шифрование GPG (ключ в env). Retention: 30 daily / 12 weekly / 24 monthly. Тест восстановления раз в неделю (cron) — поднимаем временный Postgres из последнего бэкапа, гоняем smoke. Нерабочий бэкап → блокирующий алерт.
- **Аудит-лог**: каждое state-changing API-действие → таблица `audit_log` (actor_id, action, target_id, payload_hash, ip, ua, ts).
- **Rate limiting**: 100 req/min на user-id, 1000 req/min на ip; 429 при превышении. Хранилище — Postgres (`rate_limit_buckets`) или Cloudflare Workers KV если деплой через CF.
- **Валидация на входе**: zod-схемы на КАЖДОМ Hono endpoint; ошибки → 422 + structured payload.
- **Secrets**: только в env файла `.env.local` или платформенный secret store. Никогда не в репо. Pre-commit hook сканирует на утечки (gitleaks).
- **HTTPS-only**: см. SPEC раздел про развёртывание (Cloudflare Tunnel или DuckDNS+LE).
- **CSP + HSTS + X-Frame-Options**: фронт за Hono с правильными заголовками.
- **Identity guard (paranoid)**: cookie `tg_uid` HttpOnly=false (для клиентского сравнения с initDataUnsafe) + server-side проверка cookie ↔ JWT при каждом запросе. Mismatch → инвалидация сессии + force re-auth.
- **CSRF**: double-submit token + Origin/Referer check на каждом state-changing эндпоинте.
- **Input sanitization**: текстовые поля (комментарии, отзывы) — strip-HTML, escape при выводе; `text` content-type, не `html`.
- **Sandbox-атрибуты на iframe**: при отображении любого пользовательского контента — `sandbox="allow-scripts"` нет, никаких iframe вообще для UGC.

### 5.2 Производительность
- p95 API < 200ms на чтение, < 500ms на запись.
- Лента поездок: индекс на `(status, departure_at)`; LIMIT 50 + cursor pagination.
- N+1 запрещён: review-агрегаты — materialized view `user_stats`, refresh каждые 5 минут.
- Realtime: WebSocket через Supabase, fallback на polling раз в 30 сек.
- Карта: clustering при ≥50 точках в области (Leaflet markercluster).

### 5.3 Надёжность
- Health-check endpoints `/health` и `/readiness`.
- Graceful shutdown Hono + Bun (drain in-flight requests).
- Idempotency keys на write-endpoints (header `Idempotency-Key`).
- Circuit breaker для вызовов в Telegram Bot API (при 5xx — пауза 30 сек, fallback на «push потом»).
- Retry с exponential backoff на DB connection errors.

### 5.4 Совместимость
- Telegram WebApp API ≥ 6.0 (актуальная версия на 2026-05).
- Браузеры: WebView Telegram (Android/iOS/Desktop). Standalone web — не цель.
- Темизация: подхват `Telegram.WebApp.colorScheme` (light/dark).

### 5.5 Доступность
- WCAG 2.1 AA на критичных сценариях.
- Контраст / aria-labels / клавиатурная навигация в desktop-WebApp.

### 5.6 Локализация
- v1.0 — только русский. i18n-ready через `i18next` (заложить структуру).

## 6. Технический стек (с учётом бюджета $0)

| Слой | Технология | Free tier / стоимость | Обоснование |
|------|-----------|------------------------|-------------|
| Backend | TypeScript + Hono + Bun | бесплатно | Утверждено; Hono-Web Standards совместим с Cloudflare Workers как fallback |
| Frontend | TypeScript + Vite + React (SPA) | бесплатно | Утверждено |
| БД | Supabase (PostgreSQL) | free tier 500MB БД, 2GB transfer, 50k MAU Auth, 1GB storage | достаточно для MVP в одном ЖК |
| Деплой backend | Docker Compose на личном сервере + Cloudflare Tunnel | бесплатно | без домена — туннель CF даёт `*.cfargotunnel.com` HTTPS; альтернатива: Fly.io free (3×256MB VMs) |
| Деплой frontend | Cloudflare Pages | бесплатно (unlimited bandwidth) | SPA + edge cache |
| HTTPS/Domain | Cloudflare Tunnel или DuckDNS поддомен + Let's Encrypt | бесплатно | TG MiniApp требует валидный TLS на домене; bare IP не работает |
| Карта | Leaflet + OpenStreetMap | бесплатно, без квот | без API-ключей |
| Auth | Supabase Auth + Telegram initData (custom JWT) | бесплатно (входит в Supabase free) | |
| Аналитика | PostHog Cloud free tier (1M events/мес) | бесплатно | managed, не нагружаем сервер |
| AI чат / Whisper | Anthropic / OpenAI | пост-MVP | заложено в стек, в MVP не используется |

**Платные альтернативы в roadmap при росте**:
- Supabase Pro ($25/мес) при превышении 500MB / 50k MAU.
- Railway / Fly.io paid при превышении free часов.
- PostHog paid при >1M events/мес.
- Yandex Maps API при необходимости routing/geocoding.
