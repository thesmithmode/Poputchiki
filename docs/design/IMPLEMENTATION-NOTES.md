## Что есть

  - `shell.jsx` — оболочка приложения (телефонный фрейм, навигация).
  - `ride-card.jsx` — карточка поездки (3 layout-варианта).
  - `filters.jsx` — фильтры доверия и поиска.
  - `address.jsx` — выбор адреса/направления.
  - `screens-feed.jsx` — лента поездок.
  - `screens-map.jsx` — карта с маркерами и маршрутами.
  - `screens-detail.jsx` — детальная карточка поездки.
  - `screens-profile.jsx` — публичный профиль водителя/пассажира.
  - `screens-create.jsx` — форма создания поездки.
  - `screens-rest.jsx` — оставшиеся вспомогательные экраны (онбординг, настройки и т.п.).
  - `app.jsx` — корневой компонент с роутингом между экранами.
- `project/android-frame.jsx`, `project/ios-frame.jsx` — фреймы платформ (для preview).
- `project/design-canvas.jsx` — артборд-обёртка для просмотра прототипа.
- `project/tweaks-panel.jsx` — переключатели вариаций (тема, плотность, layout, фильтры).

## Что зафиксировано в дизайне (по chat1.md)

- **Скоуп**: полный MVP-флоу (лента, карточка поездки, профиль, создание, карта, фильтры, лайки/отзывы).
- **Уровень**: production-ready максимальный.
- **Платформа**: универсально (iOS / Android / Desktop в Telegram WebApp).
- **Карта**: критична — полный мок маркеров и маршрутов через Leaflet.
- **Trust emphasis**: бейджи «новый участник», крупный счётчик лайков, фильтры доверия на видном месте.
- **Палитра / шрифты**: Telegram light/dark theme, акцент-цвет настраивается, `system-ui` шрифт-стек.
- **Данные**: реалистичные русские имена жителей ЖК Царёво, маршруты Казани (центр, метро, аэропорт, вокзал).
- **Tweaks**: переключатели темы, акцентного цвета, плотности ленты, layout карточки, тип карточки (с превью карты / без), role-view (новый / прокачанный аккаунт), фильтры по умолчанию вкл/выкл.

## Как использовать при реализации

Это **prototype, не production code**. Цель — пиксель-в-пиксель воспроизвести визуальный результат на нашем стеке (Vite + React + TS + Tailwind), не копировать внутреннюю структуру прототипа дословно.

### Правила

1. **Читай chat1.md ПЕРЕД любой UI-задачей** — там зафиксированы интенты и решения, которые не видны в коде.
2. **Читай конкретный экран целиком** перед имплементацией: `screens-feed.jsx` — для F-RIDE ленты, `screens-detail.jsx` — для F-RIDE карточки, `screens-profile.jsx` — для F-PROFILE и т.д.
3. **Следуй импортам** прототипа: `screens-feed.jsx` тянет `ride-card.jsx`, `filters.jsx`, `data.jsx`, `ui.jsx` — все нужно прочитать.
4. **Не копируй inline стили as-is** — переноси на Tailwind классы (или CSS modules / styled-components — что в стеке).
5. **Не копируй мок-данные из `data.jsx`** в код — это только для preview. Реальные данные приходят из API (`apps/api`).
6. **Иконки** — используй `lucide-react` (или похожий пакет) вместо inline SVG из `icons.jsx`.
7. **Карта** — Leaflet + React-Leaflet (см. PRD §4.3 F-MAP).
8. **Tweaks panel** не реализуется в prod — это design exploration tool. Light/dark theme подхватывается через `Telegram.WebApp.colorScheme`. Density / layout — фиксированные значения, выбранные продуктом.

### Дополнительные UI-фичи из дизайна, которые расширяют PRD

- **Schedule visualization** для регулярных рейсов — отдельная вкладка в профиле водителя (Пн–Пт 8:30 → точка).
- **Public stats grid** в профиле — % состоявшихся, среднее время ответа, месяцев в сервисе, регулярных поездок.
- **Гистограмма рейтинга** — распределение 5★/4★/3★/2★/1★ полосками.
- **Passenger list** в карточке поездки — кликабельные аватары с переходом в профиль.
- **Кнопка «Написать в Telegram»** как primary action в карточке поездки.
- **Парная кнопка лайк/контакт** — равные по весу действия.

Эти фичи можно добавить в `OPEN-QUESTIONS` или сразу в PRD при ревизии v0.4.

## Не делать

- Не рендерить HTML файл в браузере без явного запроса пользователя.
- Не скриншотить — всё нужное есть в исходниках.
- Не править файлы дизайна — это исторический срез, эталон. Менять только если ревизия дизайна запрошена пользователем.
