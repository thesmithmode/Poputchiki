# Ban Evasion: Принятые решения и мониторинг

## Проблема

Telegram предоставляет `tg_id` как единственный идентификатор пользователя. Новый Telegram-аккаунт = чистая запись в нашей БД — никаких cookies, device fingerprint, IP-binding или иной привязки, которая позволила бы отследить ранее забаненного пользователя.

**Вывод:** Полностью предотвратить ban evasion через новый TG-аккаунт — невозможно без ущерба для приватности всех пользователей.

## Принятое решение: trust-by-behaviour, not fingerprint

Вместо попытки отследить личность используем поведенческие барьеры:

| Барьер | Реализация |
|--------|-----------|
| Возраст аккаунта | `tg_account_age_days` < 30 → ограниченная активность (1 поездка/сутки, не показывается в ленте) |
| Лайки | Новый аккаунт без лайков не видит контакты других пользователей |
| Rate limits | API: 10 поездок/час от одного `tg_id`; нарушение → автоблок + admin-алерт |
| Жалобы | 3+ жалобы от разных пользователей → автоблок, review в admin-панели |

**Что мы НЕ делаем:**
- Не собираем device fingerprint, User-Agent, IP-адреса для идентификации
- Не требуем верификацию телефона или документов (пост-MVP, при масштабировании)
- Не блокируем диапазоны IP (ложные срабатывания на NAT/VPN)

## Мониторинг: anomaly-detect.ts

Ежедневный cron-джоб (`apps/cron/src/anomaly-detect.ts`) проверяет:

- **Спайк регистраций**: >50 новых пользователей за 24 часа → admin TG-алерт
  - Порог настраивается через переменную `NEW_USER_ALERT_THRESHOLD` в коде
  - Нормальный органический рост ЖК: 5–20 пользователей/сутки
  - >50 = аномалия (бот-регистрация или координированный ban evasion)

## Реакция на алерт

При получении алерта "Аномальный рост регистраций":

1. Проверить `audit_log` → события `user_created` за последние 24 часа
2. Смотреть на паттерны: схожие имена, одновременная регистрация пачками, отсутствие активности
3. Если массовая атака → временно поднять `min_account_age_days` до 90 через env
4. Подозрительные аккаунты — ручной бан через admin-панель

## Ограничения MVP

- Нет автоматического определения "это тот же человек с нового аккаунта"
- Нет ML-модели для поведенческого clustering
- Нет верификации через другие каналы (SMS, email)

Эти механизмы — пост-MVP при достижении >1000 MAU и наличии реальных инцидентов.
