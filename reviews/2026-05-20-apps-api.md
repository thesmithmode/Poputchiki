## Critical

### [C1] CORS: отсутствует fail-fast при пустом `DOMAIN` — при старте без DOMAIN весь frontend недоступен (известная проблема, НЕ исправлена)
- Files: `apps/api/src/middleware/cors.ts:8-9`
- Problem: `allowedOrigin = \`https://app.${process.env.DOMAIN ?? ""}\`` — при отсутствии `DOMAIN` строка становится `"https://app."`. Ни один реальный origin не совпадёт → CORS-заголовки не выставляются → браузер блокирует все `/api/*` запросы с CORS-ошибкой. Сервер стартует без предупреждений.
- Why bad: silently broken при каждом деплое без `DOMAIN`. Principle fail-fast нарушен. Сбой незаметен — ошибка видна только в браузере, не в логах сервера.
- Fix:
```ts
// В parseApiEnv или index.ts:
const domain = process.env.DOMAIN;
if (!domain) throw new Error("DOMAIN env var is required — cannot configure CORS");
```
- Evidence: `cors.ts:8`; описано в `.memory/knowledge/concepts/cors-startup-failsafe.md` — это **регрессия**, проблема была зафиксирована ещё 2026-05-15 и остаётся нерешённой.

### [C2] `db/crypto.ts`: `pgcryptoKey = process.env.PGCRYPTO_KEY ?? ""` — при пустом ключе PII шифруется пустой строкой без ошибки
- Files: `apps/api/src/users/usersRouter.ts:169`, `apps/api/src/db/crypto.ts:6`
- Problem: `const pgcryptoKey = process.env.PGCRYPTO_KEY ?? ""` — если `PGCRYPTO_KEY` не задан, `encryptPii` вызывается с пустой строкой как ключом. `set_config('pgcrypto.key', '', true)` выставляет пустой ключ → `pgp_sym_encrypt(plaintext, '')` → phone/apt_number либо шифруются слабым ключом, либо DB-функция падает. Нет fail-fast.
- Why bad: PII compliance — телефоны пользователей могут быть зашифрованы пустым ключом (тривиально дешифруемы) без каких-либо предупреждений в логах.
- Fix:
```ts
const pgcryptoKey = process.env.PGCRYPTO_KEY;
if (!pgcryptoKey) {
  logger.error({ event: "missing_pgcrypto_key" }, "PGCRYPTO_KEY not set — aborting PII write");
  return c.json({ error: "server misconfigured" }, 500);
}
```
- Evidence: `usersRouter.ts:169`

### [C3] `/auth/logout`: revocation JTI не атомарна — access token может жить после logout
- Files: `apps/api/src/auth/authRouter.ts:291-301`
- Problem: два INSERT в `revoked_tokens` (refresh JTI + access JTI) выполняются в отдельных `try/catch` без транзакции. Если второй INSERT упадёт — refresh token отозван, access token живёт ещё до 15 минут. Пользователь думает что вышел из системы.
- Why bad: security gap — украденный access token продолжает работать после logout.
- Fix:
```ts
await sql.begin(async (tx) => {
  for (const jti of jtiList) {
    await tx`INSERT INTO revoked_tokens (jti, user_id) VALUES (${jti}, ${userId}) ON CONFLICT DO NOTHING`;
  }
});
```
- Evidence: `authRouter.ts:290-301` — цикл `for (const jti of jtiList)` без `sql.begin`

### [C4] `DELETE /me`: анонимизация не атомарна — race condition между cancel rides и anonymize_user
- Files: `apps/api/src/users/usersRouter.ts:275-312`
- Problem: операции выполняются в трёх отдельных транзакциях/вызовах:
  1. `withIdentity(...)` — cancel rides + collect passengers (commits)
  2. `sql\`SELECT app.anonymize_user(...)\`` — вне транзакции
  3. `sql\`UPDATE revoked_tokens...\`` — вне транзакции

  Если между 1 и 2 пользователь создаёт новый ride (возможно, т.к. rides уже отменены но anonymize ещё нет) → новый ride не будет отменён. Если шаг 2 падает после шага 1 — rides отменены, пользователь НЕ анонимизирован (PII остаётся).
- Why bad: GDPR/PII compliance риск при DB blip; возможна частичная анонимизация.
- Fix: перенести `anonymize_user` вызов внутрь `withIdentity` транзакции (SECURITY DEFINER функция уже есть).
- Evidence: `usersRouter.ts:275-312`

## High

### [H1] `geocodeRouter`: `lastRequestAt` — утечка памяти при 50k DAU, не работает при нескольких репликах
- Files: `apps/api/src/geocode/geocodeRouter.ts:88`
- Problem: `const lastRequestAt = options._lastRequestAt ?? new Map<string, number>()` — module-level Map никогда не очищается. При 50k DAU растёт до 50k записей. При горизонтальном масштабировании (2+ реплики) каждая реплика — свой Map → rate limit per-user не работает.
- Fix: хранить в `rate_limit_buckets` (как остальной rate limit), или добавить TTL eviction: при обращении удалять записи старше 5 минут.
- Evidence: `geocodeRouter.ts:88`

### [H2] `banned-user`: in-memory кэш не синхронизируется между репликами
- Files: `apps/api/src/middleware/banned-user.ts:18`
- Problem: `const cache = new Map<...>()` — module-level singleton. Admin банит пользователя → `invalidateUserState(userId)` срабатывает только на обработавшей реплике. На других репликах кэш TTL 30s продолжает пускать забаненного.
- Why bad: при масштабировании на 50k пользователей ban enforcement не работает. Это **известная проблема** (`.memory/knowledge/concepts/banned-user-cache-invalidation.md`) — в памяти зафиксирован факт: "cache is process-local — multiple API replicas have independent caches; 30s TTL means max lag across replicas is 30s". Но 30s при одной реплике ≠ 30s при нескольких.
- Fix: при добавлении второй реплики — Redis pub/sub для инвалидации между репликами.
- Evidence: `banned-user.ts:18`

### [H3] `ridesRouter GET /`: `cache.clear()` при каждой мутации — O(users × queries) thrash при 50k DAU
- Files: `apps/api/src/rides/ridesRouter.ts:311,395,731,842,907`
- Problem: `cache.clear()` при каждом create/cancel/complete/patch очищает весь кэш для всех пользователей. При 50k DAU с высокой write-частотой кэш практически никогда не используется — его TTL < write interval.
- Why bad: кэш с per-user ключом и global clear = неэффективен при нагрузке.
- Fix: инвалидировать только ключи, начинающиеся с `${user.id}:`, или использовать версионный ключ (`rides_version` counter) для tag-based invalidation.
- Evidence: `ridesRouter.ts:311` `cache.clear()`

### [M4] `db/crypto.ts`: `set_config('pgcrypto.key', key, true)` — ключ PII устанавливается в session GUC, виден другим запросам в том же коннекшене если нет `SET LOCAL`
