## Critical

### [C1] encrypt_pii помечена STABLE — семантически неверно и небезопасно

- Files: `db/migrations/006_pgcrypto_pii.sql:9`, `db/migrations/006_pgcrypto_pii.sql:21`
- Problem: `app.encrypt_pii` и `app.decrypt_user_pii` объявлены `STABLE`, хотя они вызывают
  `pgp_sym_encrypt` / `pgp_sym_decrypt` (недетерминированны по IV — каждый вызов даёт
  другой зашифрованный блок), и читают `current_setting('pgcrypto.key')` — значение сессии.
  `STABLE` позволяет планировщику кешировать результат между вызовами внутри одного
  statement; для `encrypt_pii` это означает, что два вызова с одним plaintext
  могут вернуть один и тот же ciphertext (кешированный), нарушая семантику PGP. Кроме
  того, `STABLE` некорректно для функций, которые читают session GUC — правильный
  volatility `VOLATILE`.
- Why bad: IND-CPA-безопасность pgp_sym_encrypt строится на уникальном IV; кеширование
  его нарушает. Функция зависит от GUC (не от аргументов) — кеширование неверно.
- Fix: заменить `LANGUAGE sql STABLE SECURITY DEFINER` на
  `LANGUAGE sql VOLATILE SECURITY DEFINER` в обоих функциях.
- Evidence: `006_pgcrypto_pii.sql:9`, `006_pgcrypto_pii.sql:21`.

### [C2] revoked_tokens без FORCE ROW LEVEL SECURITY

- Files: `db/migrations/008_revoked_tokens.sql:8–9`
- Problem: `ALTER TABLE revoked_tokens ENABLE ROW LEVEL SECURITY` выполнен, но
  `FORCE ROW LEVEL SECURITY` отсутствует. Владелец таблицы (суперпользователь, создающий
  схему) обходит RLS по умолчанию. Если соединение к БД выполняется от суперпользователя
  или владельца — политика `RESTRICT USING (false)` не применяется, и он может читать
  все отозванные токены или вставлять произвольные записи.
- Why bad: `revoked_tokens` — критическая таблица безопасности; без FORCE RLS её
  содержимое видно привилегированным соединениям. Контраст: все остальные таблицы
  имеют `FORCE ROW LEVEL SECURITY`.
- Fix: добавить `ALTER TABLE revoked_tokens FORCE ROW LEVEL SECURITY;` после
  строки 8 в `008_revoked_tokens.sql`.
- Evidence: все 25 других таблиц в мигрциях 001–028 имеют FORCE RLS;
  `revoked_tokens` — единственное исключение.

### [H2] user_stats MV не имеет RLS, раскрывает агрегаты всех пользователей

- Files: `db/migrations/010_user_stats_mv.sql`, `db/migrations/023_split_avg_stars.sql`
- Problem: `user_stats` — материализованное представление без `ENABLE ROW LEVEL
  SECURITY`. Любой запрос от `poputchiki_app` (обычный аутентифицированный пользователь)
  может сделать `SELECT * FROM user_stats` и получить агрегаты всех пользователей:
  `rides_as_driver_completed`, `likes_received`, `avg_stars`. RLS на MV в Postgres
  должна быть задана явно через `ALTER MATERIALIZED VIEW ... ENABLE ROW LEVEL SECURITY`.
  Без неё данные видны всем, у кого есть SELECT-грант по DEFAULT PRIVILEGES.
- Why bad: `avg_stars` и количество поездок — косвенно идентифицирующие PII
  данные. Ни одному рядовому пользователю не нужны агрегаты других.
- Fix: PostgreSQL 16 не поддерживает RLS на материализованных представлениях —
  `ALTER MATERIALIZED VIEW ... ENABLE ROW LEVEL SECURITY` не существует. Корректный
  подход — отозвать прямой SELECT и выставить данные через SECURITY INVOKER view:
  ```sql
  -- В миграции 029:
  REVOKE SELECT ON user_stats FROM poputchiki_app;
  CREATE VIEW user_stats_view
    WITH (security_invoker = true) AS
    SELECT * FROM user_stats
    WHERE user_id = app.current_user_id() OR app.is_admin();
  GRANT SELECT ON user_stats_view TO poputchiki_app;
  ```
  Либо, если нужен доступ сервисного уровня — читать через функцию
  `SECURITY DEFINER` с явной проверкой прав.
- Evidence: `010_user_stats_mv.sql:1–19`, `023_split_avg_stars.sql:49–69`.
  DEFAULT PRIVILEGES в `000_app_identity.sql:41–42` дают SELECT всем таблицам,
  включая MV.

### [H4] notif_update policy на user_notifications без WITH CHECK

- Files: `db/migrations/025_user_notifications.sql:31–32`
- Problem: `CREATE POLICY notif_update ON user_notifications FOR UPDATE
  USING (user_id = app.current_user_id())` — отсутствует `WITH CHECK`. Без
  `WITH CHECK` пользователь может обновить свою запись и изменить `user_id` на
  чужой (перенести уведомление на другого пользователя). FORCE RLS применяет
  только USING к результирующим строкам, но не проверяет новые значения.
- Why bad: пользователь A может переназначить своё уведомление пользователю B,
  затем B видит это уведомление как своё.
- Fix:
  ```sql
  CREATE POLICY notif_update ON user_notifications
    FOR UPDATE
    USING (user_id = app.current_user_id())
    WITH CHECK (user_id = app.current_user_id());
  ```
- Evidence: `025_user_notifications.sql:31–32`. Для сравнения, корректный паттерн
  есть в `002_rides.sql:117–126` (ride_requests_update), `003_social.sql:84–85`.

### [H8] enqueueNotification делает два await sql в шеринге транзакции — нарушение атомарности при ошибке pg_notify

- Files: `packages/shared/src/notifications/enqueue.ts:42–58`
- Problem: функция делает `await sql INSERT INTO user_notifications`, затем
  `await sql SELECT pg_notify(...)`. Если `pg_notify` упадёт (например, payload
  > 8000 байт — лимит Postgres NOTIFY), строка в `user_notifications` уже
  вставлена и зафиксирована (если `sql` — не TransactionSql). Если вызывается
  внутри транзакции (TransactionSql), то ошибка pg_notify откатит и INSERT, и
  всю внешнюю транзакцию — что может означать откат бизнес-операции (принятие
  заявки) из-за сбоя уведомления.
- Why bad: `pg_notify` payload не валидируется; строка >8000 байт реальна при
  больших полях `data`. Сбой уведомления не должен откатывать бизнес-транзакцию.
- Fix: обернуть `pg_notify` в отдельный блок с обработкой ошибки, не давая ей
  всплыть; либо валидировать длину payload до вызова:
  ```ts
  if (payload.length > 7500) {
    // обрезать data или пропустить notify, строка в БД уже есть
    return;
  }
  await sql`SELECT pg_notify('notify_user', ${payload})`;
  ```
- Evidence: `enqueue.ts:42–58`. Предел 8000 байт — официальный лимит PostgreSQL
  LISTEN/NOTIFY.

### [H9] ride_requests не имеет DELETE политики — пассажир не может удалить заявку

- Files: `db/migrations/002_rides.sql:103–126`
- Problem: для `ride_requests` есть политики READ, INSERT, UPDATE — но нет DELETE
  политики. Отзыв заявки пассажиром реализуется через UPDATE статуса (`cancelled`),
  что ок. Но отсутствие DELETE policy означает, что если код когда-либо попытается
  DELETE ride_request (например, в каскаде или admin cleanup), FORCE RLS его
  заблокирует с 0 rows, а не ошибкой — тихое игнорирование операции.
  Дополнительно: это расхождение с ride_templates, у которого есть явная DELETE policy.
- Why bad: тихое игнорирование DELETE без ошибки. Потенциальный data leak если
  admin-cleanup ожидает удаления строк.
- Fix: добавить явную DELETE policy или задокументировать что DELETE запрещён намеренно:
  ```sql
  -- Запрещено явно: заявки только отзываются через UPDATE status='cancelled'
  -- DELETE policy намеренно отсутствует (FORCE RLS = deny-by-default)
  ```
  Либо добавить admin-only policy:
  ```sql
  CREATE POLICY ride_requests_admin_delete ON ride_requests
    FOR DELETE USING (app.is_admin());
  ```
- Evidence: `002_rides.sql:103–126`.

### [M9] rides UPDATE policy не позволяет admin'у изменять поездки — нет admin override

### [M10] sanitize/index.ts экспортирует внутренние хелперы (stripHtml, normalizeWhitespace, limitLength)

### [M11] complaint_week_utc в public схеме без SET search_path — функциональный индекс уязвим

- Files: `db/migrations/003_social.sql:137–148`
- Problem: `complaint_week_utc` — `LANGUAGE sql IMMUTABLE`, не SECURITY DEFINER,
  вызывается из функционального уникального индекса на `complaints`. Функция живёт
  в `public` схеме, без `SET search_path`. Без `SET search_path = pg_catalog`
  пользователь с правом CREATE в своей схеме может создать `date_trunc` в ней —
  при правильно выстроенном search_path он будет вызван вместо системного, и индекс
  вернёт искажённые значения, нарушив антиспам-защиту. Нарушает и project convention:
  все utility-функции живут в `app`.
- Why bad: uniqueness constraint недельных жалоб перестаёт работать → один пользователь
  может подать N жалоб в неделю. Вектор нетривиальный (требует CREATE SCHEMA прав),
  но последствие серьёзное.
- Fix:
  ```sql
  CREATE OR REPLACE FUNCTION app.complaint_week_utc(ts timestamptz)
  RETURNS timestamp LANGUAGE sql IMMUTABLE
  SET search_path = pg_catalog AS $$
    SELECT date_trunc('week', (ts AT TIME ZONE 'UTC'));
  $$;
  -- DROP старый индекс и пересоздать с app.complaint_week_utc
  ```
- Evidence: `003_social.sql:137–148`.
