# Улучшения для ralph.sh

## 1. Error Handling — КРИТИЧНО
- Проверять статус выхода команд (tests, lint)
- Если тест упал — выводить ошибку и **ОСТАНАВЛИВАТЬ** цикл
- Или добавить retry механизм: дать Claude возможность переделать задачу если она упала

## 2. Logging и Tracking
- Сохранять output каждой итерации в отдельный файл (`logs/iteration-1.txt`, `iteration-2.txt`, и т.д.)
- Так если что сломалось, можно посмотреть что случилось в истории
- Сейчас всё выводится в консоль и теряется после скрола

## 3. Timeout на выполнение
- Что если Claude зависнет на одной задаче на час?
- Добавить timeout: если итерация длится > 10 мин, убивать процесс и переходить дальше
- Настраиваемый лимит через `RALPH_TIMEOUT` переменную

## 4. Backup перед итерацией
- Перед каждой итерацией делать git stash / commit с автосохранением
- Если Claude накосячил, откатываться легко командой `git reset --hard`
- Создавать checkpoint перед рискованными операциями

## 5. Уведомления о статусе
- Discord webhook или Telegram боту писать: "итерация 5, задача 50% готова"
- Отправлять уведомление если итерация упала с ошибкой
- Сейчас только `say` команда, которая не работает везде (macOS only)

## 6. Более детальный prompt для Claude
- Текущий prompt слишком генерик
- Добавить контекст: "ты видишь эти файлы: ...", "последняя итерация сломала ...", "будь внимательнее с X"
- Включать git diff с последними коммитами для контекста
- Напоминать о наиболее частых ошибках

## 7. Dependency Check перед началом
- Проверять что `claude` CLI установлен и работает
- Проверять что `uv` работает
- Проверять что git готов (нет незакомиченных изменений)
- Проверять что `.env` файл существует и имеет нужные переменные
- Не ждать 30 сек пока упадёт на первой же задаче

## 8. Post-Mortem Analysis после каждой итерации
- Если нашлась ошибка, Claude должен её анализировать
- Добавить специальный prompt: "Что случилось? Почему упал тест? Как это исправить?"
- Сохранять анализ ошибок в `error-log.md`

## 9. Performance Monitoring
- Выводить время выполнения каждой итерации
- Показывать средний time per task
- Предупреждать если итерация выполняется аномально долго

## 10. Better Status Display
- Progress bar вместо просто счётчика
- Показывать: сколько задач done, сколько pending, сколько осталось
- Выводить estimated time to completion

## 11. Graceful Shutdown
- Обработка Ctrl+C (SIGINT)
- При прерывании: сделать финальный commit, сохранить прогресс, вывести summary
- Не оставлять repo в broken state

## 12. Configuration File
- Создать `ralph.config.json` или `ralph.yml` для настроек:
  - `max_iteration_time`: таймаут на одну итерацию
  - `max_retries`: сколько раз переделывать упавшую задачу
  - `log_level`: verbose, normal, quiet
  - `notifications`: включить/выключить Discord/Telegram
  - `auto_commit`: автокоммитить ли после каждой итерации

## 13. Task Filtering
- Позволить запустить скрипт для конкретной задачи: `./ralph.sh --task TASK-005`
- Позволить пропустить задачи: `./ralph.sh --skip TASK-001,TASK-002`
- Позволить приоритизировать категории: `./ralph.sh --category infrastructure`

## 14. Git Integration Improvements
- Использовать git tags для отметки завершённых итераций: `iteration-1`, `iteration-2`
- Сохранять ветку для каждой итерации: `iteration/1`, `iteration/2`
- Проще откатываться и смотреть историю

## 15. Agent Communication
- Дать агенту доступ к истории предыдущих ошибок
- Добавить в prompt список "что не работало до этого"
- Claude будет учиться на своих ошибках

# Проверки архитектуры, паттернов, принципов (КРИТИЧНО)

## 16. Architecture Compliance Check
- Перед коммитом проверять что код соответствует архитектуре проекта
- Парсить файловую структуру и проверять:
  - Импорты идут только вниз по слоям (не вверх, не в сторону)
  - Domain logic не смешана с infrastructure
  - Handlers не содержат бизнес-логику
- Выводить предупреждение если Claude нарушает слои

## 17. Code Quality Anti-Patterns Detection
- Скрипт должен сканировать код на common anti-patterns:
  - **God objects** (классы > 300 строк) → предупреждение
  - **Long methods** (методы > 50 строк) → split required
  - **Circular dependencies** → ERROR, не коммитить
  - **Duplicate code** (более 10 одинаковых строк) → refactor
  - **Magic numbers** (числовые константы в коде) → extract to constants
  - **Type hints missing** → проверять что все параметры типизированы

## 18. Dependency Injection Check
- Проверять что классы не создают свои зависимости через `new`
- Все зависимости должны инжектиться через constructor или factory
- Если нарушение — выводить ошибку и не коммитить

## 19. SOLID Principles Validator
- **S (Single Responsibility):** класс должен иметь одну причину для изменения
  - Если класс делает несколько вещей → warning
- **O (Open/Closed):** открыт для расширения, закрыт для изменения
  - Проверять что новая фича не меняет старый код, а расширяет
- **L (Liskov):** подклассы должны быть заменяемы на базовые
- **I (Interface Segregation):** не создавать толстые интерфейсы
  - Если интерфейс > 5 методов → мб надо его разбить
- **D (Dependency Inversion):** зависить от абстракций, не от конкреции

## 20. Test Coverage Enforcer
- Требовать чтобы новый код имел тесты
- Если функция добавлена, но для неё нет теста → не коммитить
- Проверять что покрытие не упало ниже заданного порога (e.g., 80%)
- Выводить: какие строки кода uncovered

## 21. Security Check
- Сканировать на типичные security issues:
  - Hardcoded secrets (API keys, passwords) → ERROR
  - SQL injection points → warning
  - Unvalidated user input → warning
  - Logging sensitive data → warning
  - Missing authentication/authorization → error
- Использовать simple regex patterns или integrated tools

## 22. Documentation Enforcer
- Проверять что новые public функции имеют docstrings
- Если функция without documentation → warning, но не блокировать
- Если публичный API изменился → требовать update README.md
- Проверять что CHANGELOG обновлён

## 23. Database Migration Check
- Если меняется schema в БД:
  - Проверять что migration написана
  - Migration должна быть reversible (должен быть rollback)
  - Не позволять делать миграции которые теряют данные без явного подтверждения
- Сохранять все миграции в версионный контроль

## 24. Performance Regression Detection
- Если код добавляет новые циклы, DB queries, API calls:
  - Предупреждать что это может быть N+1 проблема
  - Если нет индекса на column в WHERE — warning
  - Если query без LIMIT на большой таблице — error
- Не блокировать, но обязательно выводить warning

## 25. Naming Conventions Enforcer
- Проверять consistency в именовании:
  - Variables: snake_case
  - Classes: PascalCase
  - Constants: UPPER_SNAKE_CASE
  - Private methods: _leading_underscore
- Если нарушение — выводить и требовать исправления

## 26. Dead Code Detection
- После завершения задачи: сканировать на unused imports, переменные, функции
- Удалять автоматически или выводить list что удалить

## 27. Semantic Commit Messages Validator
- Commits должны быть в формате: `type(scope): message`
  - type: feat, fix, refactor, docs, test, chore
  - scope: component name
  - message: что именно изменилось
- Если commit не соответствует формату → warning (можно allow-list)

## 28. Revert-Ability Check
- Перед коммитом проверять что коммит можно откатить
- Если коммит содержит нарушения которые невозможно откатить → блокировать
- Сохранять backup ветки перед каждым большим коммитом

## 29. Breaking Changes Detection
- Если меняется public API (функции, классы, интерфейсы):
  - Требовать bump semver
  - Требовать MIGRATION guide в документации
  - Предупреждать что это breaking change

## 30. Complexity Metrics
- Использовать cyclomatic complexity для измерения сложности функций
- Если функция имеет complexity > 10 → требовать рефакторинга
- Показывать метрики в progress report

## 31. Concurrency & Race Conditions
- Если код добавляет многопоточность или async операции:
  - Проверять наличие race conditions (shared state, locks)
  - Проверять deadlock-возможности
  - Требовать tests для concurrent scenarios
- Предупреждать об использовании глобальных переменных в многопоточном контексте

## 32. Database Transactions & ACID
- Если появляются DB операции в одной фаза кода:
  - Проверять что они завёрнуты в transactions
  - Проверять isolation level и concurrency implications
  - Если бизнес-логика требует atomicity — требовать transaction
  - Предупреждать если N операций выполняются без гарантии atomicity

## 33. Idempotency Validation
- Если Claude добавляет side-effects (email, API calls, DB writes):
  - Требовать что операция может быть safely повторена
  - Если операция NOT idempotent → требовать механизма deduplication (idempotency key)
  - Проверять что retry-логика не будет создавать duplicates

## 34. Temporal Coupling Detection
- Предупреждать если код полагается на порядок операций неявно
- Если функция A должна вызваться ДО функции B — это должно быть явным
- Проверять что dependencies между операциями задокументированы

## 35. State Machine Validation
- Если добавляется stateful логика (workflows, transitions):
  - Проверять что все возможные transitions определены
  - Проверять что нет orphaned states (состояния откуда невозможно выйти)
  - Требовать diagram или явный list transitions
  - Предупреждать если state может быть invalid (e.g., completed && pending одновременно)

## 36. Backward Compatibility Check
- Если меняется API или формат данных:
  - Проверять что старые клиенты будут работать
  - Требовать migration path или deprecation period
  - Если изменение TRULY breaking — требовать major version bump и явный BREAKING CHANGES doc

## 37. Resource Leak Detection
- Сканировать на file handles, connections, memory утечки:
  - Файлы что открыты но не закрыты (try-finally или context managers)
  - DB connections что не returned в пул
  - Listeners что не unsubscribed
  - Timers что не cancelled
- Требовать cleanup в error paths

## 38. Error Message Quality Check
- Проверять что ошибки информативны (не просто "Error" или "null")
- Если исключение выбрасывается — требовать контекста (что сломалось, почему, как исправить)
- Предупреждать если error message expose внутренние детали или sensitive данные

## 39. Observability & Instrumentation
- Проверять что critical paths имеют logging/metrics/tracing
- Если добавляется async операция без logging — warning
- Требовать structured logging (не просто текстовые сообщения)
- Проверять что errors логируются с достаточным контекстом

## 40. Contract & Protocol Validation
- Если код интегрируется с external API / service:
  - Проверять что contract задокументирован (OpenAPI, protobuf, JSON schema)
  - Проверять что validation schema есть для request/response
  - Требовать explicit error handling для API failures
  - Предупреждать если assumption на API поведение не валидирована

## 41. Configuration Management
- Если добавляются config params / environment variables:
  - Требовать schema/validation для конфига
  - Проверять что все required params задокументированы
  - Требовать defaults если разумны
  - Предупреждать если config может быть в inconsistent state

## 42. Disaster Recovery & Failover
- Если добавляется критичный компонент:
  - Проверять что есть backup/fallback strategy
  - Если component может fail — должен быть graceful degradation
  - Требовать restore/recovery procedure
  - Предупреждать если нет way to recover from partial failures

## 43. API Versioning Strategy
- Если публичный API меняется:
  - Требовать version в URI или header (v1, v2, etc.)
  - Не позволять breaking changes без major version bump
  - Требовать deprecation notice перед удалением старого API
  - Проверять что old versions supported параллельно (за указанный период)

## 44. Metrics & Monitoring Consistency
- Если Claude добавляет метрики — они должны быть consistent с существующими
- Naming: как именуются метрики (snake_case, camelCase, dots)
- Cardinality check: если metric имеет labels (например, `request_duration[endpoint, method]`) — проверять что cardinality bounded (не NxM labels что растут без лимита)
- Требовать что metrics documented (что означает, как интерпретировать)

## 45. Health Check & Readiness Probe
- Если добавляется service / component:
  - Требовать `/health` endpoint (или equivalent)
  - Требовать `/readiness` если есть dependencies (DB, cache, etc.)
  - Liveness vs Readiness distinction должна быть ясна
  - Проверять что health check не дорогой в вычислении

## 46. Graceful Degradation & Circuit Breaker
- Если код зависит от external service:
  - Требовать circuit breaker паттерн
  - Если external service недоступна — должен быть fallback
  - Не должно быть infinite retries без backoff
  - Проверять timeout values reasonable (не 60 сек)

## 47. Data Validation Boundary
- Требовать что external input валидируется ПРИ ВХОДЕ
- Не внутри функций, а на boundary (API handler, message consumer, etc.)
- Если validation правило нарушено — должен быть явный error (не silent fail)
- Проверять что validation covers все constraints (length, type, format, range)

## 48. Distributed Tracing Context
- Если добавляется async / multi-service логика:
  - Требовать correlation ID / trace ID propagation
  - Если Claude добавляет новую async operation — она должна carry trace context
  - Требовать что все logs в одной trace tagged с одним ID
  - Проверять что trace context not leaked between requests

## 49. Billing & Quota Check
- Если код может trigger external APIs (LLM, email, SMS, etc.):
  - Требовать rate limiting / quota checks ПЕРЕД вызовом
  - Проверять что нет случайного DDoS на external API
  - Требовать circuit breaker if quota exceeded
  - Предупреждать если операция может быть дорогой

## 50. Cache Invalidation Strategy
- Если добавляется кэширование:
  - Требовать явную стратегию инвалидации (TTL, event-based, manual)
  - Если cache может быть stale — должен быть mechanism для refresh
  - Проверять что cache key уникален для всех вариаций данных
  - Предупреждать если cache может grow unbounded

## 51. Atomicity of Multi-Step Operations
- Если Claude выполняет sequence операций где partial success = broken state:
  - Требовать что либо all-or-nothing либо explicitly handle partial success
  - Если операция 1 из 5 упала — остальные должны откатиться или быть idempotent
  - Проверять что нет "orphaned" state если process dies на шаге 3 из 5

## 52. Feature Flag & Gradual Rollout
- Если добавляется значительное изменение:
  - Требовать feature flag для gradual rollout (не вся аудитория сразу)
  - Проверять что flag можно выключить без redeploy
  - Требовать monitoring что flag работает (можно ли отследить % users с feature enabled)

## 53. Backward Incompatible Data Format
- Если меняется format в БД, JSON, protobuf, etc.:
  - Требовать migration script
  - Требовать что migration can be rolled back
  - Проверять что new format reader может read old format (если reasonable)
  - Предупреждать если migration может take > N minutes на production
